  • 分层测试
    • UI
      • 在用户界面进行操作完成测试
      • 黑盒
        • 功能
        • 系统
      • 分析
        • 基本理念1：不稳定
          • UI经常改，UI测试用例复用率低
        • 基本理念2：低效
          • 效率低，发现BUG能力弱
        • 基本理念3: 高成本
          • 代码和分支的覆盖率提高难度大
          • 发现BUG后修复成本高
        • 基本理念4：容易实施
          • 不需要代码能力，从业人员多容易推广和实施
    • Server
      • 灰盒
        • 接口
        • 集成
      • 模块间的调用关系
      • 分析
        • 基本理念1：稳定性高
          • 接口相对稳定，用例复用率高，处于UI层和unit层之间
        • 基本理念2：中等
          • 由于没有表示层的混淆作用，效率较高，发现BUG能力较高
        • 基本理念3: 较高的成本
          • 由于接口逻辑集中，测试成本非常低，但由于有server框架混淆，定位BUG和修复成本比UI低，但别Unit高
        • 基本理念4：容易实施
          • 实施较为方便，较Unit容易推广，而且较Unit容易实现自动化
    • Unit
      • 白盒/黑盒
        • 单元
        • 单元
      • 分析
        • 基本理念1：稳定性高
          • 稳定性很高，用例几乎不用变
        • 基本理念2：效率高
          • 不存在混淆，精度高，发现BUG能力极强
        • 基本理念3: 低成本
          • 定位BUG和处理BUG成本低
        • 基本理念4：容易实施
          • 实施较为方便，较Unit容易推广
  • 自动化测试
    • 每一层都应该开展自动化测试
    • UI层
      • 工具
        • QTP
        • Selenium
      • 功能自动化测试
      • 使用工具模拟用户在界面上的操作，代替重复手工测试
      • 用作回归测试和大数据量测试
    • Unit层
      • 工具
        • java:JUnit\JTest
        • python:Unittest
        • c/c++ c/c++Test
        • 编译器自带的插件测试语法
      • 使用工具测试代码的正确性和语法正确性、规范性
      • 用作单元测试，自动化大多只用作自动提交测试报告
    • Server层
      • 工具
        • Postman
        • Fiddler
        • JMeter
      • 主要用于集成测试，自动化大多用于回归测试
